# Discovery-Bearing Branches

## Purpose

This note defines how discovery should relate to continuity without collapsing the substrate into a registry-first or topic-first system.

The goal is to let observers derive bounded discovery join sets from situated branch structure before introducing asserted registries or generalized indexing layers.

## Current doctrinal rule

Discovery should be branch-derived before it is registry-derived.

The current line is:

- continuity-bearing branches remain primary
- some branches are discovery-bearing because their structure determines where discovery should happen
- discovery topics are derived projections over those branches
- discovery projections are lossy rendezvous surfaces, not continuity itself

This means:

- branch continuity is still the substrate
- topic membership is not truth
- topic match is not equivalence
- topic match is not compatibility
- topic match is not shared identity

## Discovery-bearing versus continuity-bearing

The repository should not treat every branch as equally relevant to discovery.

Some branches may be continuity-bearing without being discovery-bearing.

The current lean is:

- `context` branches are discovery-bearing by default
- `portal` branches are discovery-bearing when they expose external contexts
- `observer` branches may be discovery-bearing when they carry explicit situated context
- `referent` branches may become discovery-bearing only when their discovery relevance is explicit and stable enough

The repository should not assume:

- every support branch deserves discovery participation
- every threshold-bearing process deserves its own topic
- every branch id should become a transport topic

## Discovery should be a join set, not one topic

An observer or active branch should derive a bounded set of discovery topics from its current situation.

The default join set should be computed in this order:

1. primary situated context
2. enclosing context chain
3. portal-visible contexts
4. declared adjacency
5. optional coarse concern overlays
6. optional direct known-peer links

This means discovery is not one universal topic. It is a small deterministic set of projections over the current situated structure.

## Containment versus exposure

Containment and exposure should remain distinct.

The current line is:

- containment is hierarchy-like
- portal exposure is graph-like
- adjacency is declared neighborhood rather than containment

So:

- primary and enclosing contexts may form a tree or chain
- portal-visible contexts should not be collapsed into that tree
- adjacency should not silently become parenthood

The repository should derive discovery from all three where relevant:

- containment
- exposure
- adjacency

without pretending they are the same relation.

## Topic posture

A topic should be treated as a deterministic rendezvous projection.

It should be derived from:

- a projection version
- a topic kind
- a basis-relative scope
- a discovery-bearing anchor
- optional quantized positioning or concern overlays

But it should not be treated as:

- full branch continuity
- a universal similarity key
- proof of compatibility
- proof of co-reference

Topics carry discovery posture, not ontology authority.

## Direct peer posture

When a peer is already known, direct peer transport is allowed and often preferable.

The current line is:

- topic discovery is not required when a relevant peer key is already known
- direct peer linkage is a transport concern, not a continuity concern
- known-peer linkage does not imply shared authorship or shared identity

This keeps discovery flexible without requiring every exchange to pass through topic discovery.

## Registry posture

The repository should avoid registry-first discovery as the default.

The current line is:

- derive discovery from branch structure first
- use coarse projection overlays second
- use asserted registries only when structure and projection are insufficient

This reduces dependency on volatile discovery registries while preserving an explicit fallback for cases where structure is too weak, too private, too unstable, or too cross-cutting.

## What remains open

The repository does not yet claim:

- a final discovery-bearing branch taxonomy
- a final topic derivation schema
- a final quantization or bucketization strategy
- a final rule for concern overlays
- a final policy for privacy-preserving discovery projection
- a final rule for when referents become discovery-bearing

## Working rule

Treat discovery topics as bounded, versioned, basis-relative projections over discovery-bearing branches.

Branches carry continuity.
Artifacts carry inspectable claims.
Topics carry discovery projections.
