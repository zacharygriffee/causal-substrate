# Post-Lab Synthesis

This document captures what the completed local lab program established, what it pressured, and what it did not settle.

## Lab Program Status

The initial ten-lab program is complete.

Completed labs:

- wake/sleep continuity
- referent persistence across absence
- primary context and directional portal
- degraded basis re-identification
- trigger to happening clustering
- boundary ambiguity
- two-portal mutual exposure
- local artifact emission
- split/merge pressure
- multi-observer non-agreement

## What The Labs Proved

### 1. The current kernel is sufficient for local doctrine pressure

The in-memory kernel is enough to represent the full planned local lab set without decentralized mechanisms.

This includes:

- segmented continuity
- carried nuclei
- lineage edges
- explicit referent bindings
- context containment
- directional portals
- derived views
- persistence estimates

### 2. The current working defaults are viable

The labs support the current defaults as workable implementation posture:

- one primary situated context
- directional portals
- `continuing` / `ambiguous` / `broken` persistence states
- local source continuity with explicit emitted artifacts

### 3. Ambiguity can be represented honestly

The lab program repeatedly showed that ambiguity is not failure. It is often the correct state:

- ambiguous referent persistence
- ambiguous boundary placement
- non-agreement across observers

This is important because it validates the repo’s refusal to force identity, convergence, or agreement too early.

### 4. Lineage-first handling holds up under pressure

Split and merge pressure are still representable without collapsing into a universal identity rule.

The labs show that:

- ancestry relations can remain explicit
- carried nucleus can remain separate from ancestry
- merge pressure can be represented without pretending semantics are fully solved

### 5. Portal doctrine is locally viable

The labs showed that directional portals are enough for:

- constrained external exposure
- mutual exposure through paired portals
- preserving asymmetry between directions

This supports keeping portal doctrine directional by default.

### 6. Local-first artifact emission is already possible

The current kernel can already emit explicit local artifacts such as:

- bindings
- lineage claims
- views
- state estimates

with provenance carried as metadata, while raw continuity remains local.

## What The Labs Pressured

The labs did not close every doctrine question. They made the remaining pressure points more concrete.

### 1. Branch continuation thresholds

The model can represent continuation and branch split pressure, but it still does not decide when a continuation should become a new branch.

### 2. Multi-nucleus meaning

The model can represent multiple inherited nuclei on a successor continuity, but doctrine still does not define what that composition means.

### 3. Boundary ambiguity structure

Boundary ambiguity can be represented, but it remains unclear whether it should stay metadata-plus-view, become a stronger artifact, or trigger continuity pressure more directly.

### 4. Portal interface structure

Paired directional portals work, but the repo still has not decided whether some higher-order interface or membrane artifact should group them without implying symmetry.

### 5. Provenance strength

Provenance works as metadata for now, but the repo still needs to decide when provenance should become a stronger typed contract or explicit artifact wrapper.

### 6. Comparability and compatibility artifacts

The lab program showed non-agreement can be represented cleanly, but comparability and compatibility are still represented indirectly rather than as first-class artifacts.

## What The Labs Did Not Try To Prove

The lab program did not attempt to settle:

- decentralized exchange protocols
- mandatory convergence logic
- global truth semantics
- final basis algebra
- final receipt format
- final comparison artifact schema

These remain future work by design.

## Overall Conclusion

The repo has crossed an important threshold.

It no longer has only doctrine text and a minimal kernel. It now has an executable local lab program that validates the current defaults and exposes the remaining open questions in concrete form.

That means the next phase should not be more abstract doctrine alone. The next phase should be targeted synthesis and prototype work informed by the specific pressure points the labs exposed.

Current implementation direction after that threshold:

- keep the in-memory kernel as ontology scaffold
- bias the first serious substrate lab toward Corestore and Hypercore
- defer Hyperbee until indexing pressure is concrete
