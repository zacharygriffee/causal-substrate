# Doctrine Roadmap

## Phase 1: Ontology Freeze

Deliver:

- concept definitions
- non-goals
- forbidden collapses
- open questions
- minimal repository commitments

Exit condition:

Core terms are stable enough that later code does not silently redefine them.

## Phase 2: Structural Kernel

Deliver:

- minimal primitives
- allowed relations
- invariants
- required vs optional structure

Exit condition:

The kernel can represent observer-relative continuity, referent tracking, context containment, portal exposure, wake/sleep segmentation, and inheritance.

## Phase 3: Operational Semantics

Deliver:

- segment open/close doctrine
- trigger to happening doctrine
- carried nucleus doctrine
- binding doctrine
- context membership doctrine
- portal exposure doctrine

Exit condition:

Example scenarios can be represented without inventing hidden rules ad hoc.

## Phase 4: Epistemic Doctrine

Deliver:

- observer-relative truth posture
- basis doctrine
- identity vs lineage doctrine
- comparability/compatibility/equivalence/convergence distinctions
- agreement doctrine

Exit condition:

The repository can clearly state what it claims and what it refuses to claim.

## Phase 5: Temporal Doctrine

Deliver:

- wake/sleep doctrine
- chunking doctrine
- sealed segment doctrine
- inertia/volatility doctrine
- negative evidence doctrine

Exit condition:

Continuity under absence and resumption is representable without naive assumptions.

## Phase 6: Context Doctrine

Deliver:

- context containment semantics
- situatedness doctrine
- portal membrane doctrine
- local vs exposed structure rules

Exit condition:

Contexts and portals are structurally meaningful rather than descriptive wrappers.

## Phase 7: Exchange Doctrine

Deliver:

- local vs shared artifact classes
- portable anchors, bindings, receipts, or views
- backend preservation requirements

Exit condition:

Exchange and persistence backends can be added without rewriting the ontology.

## Phase 8: Reference Scenarios

Deliver:

- continuity across wake/sleep
- ambiguous re-identification
- split/merge lineage case
- portal-mediated outside awareness
- degraded basis case
- multi-observer non-agreement case

Exit condition:

Doctrine survives concrete examples without contradiction or premature narrowing.

## Phase 9: Implementation Alignment

Deliver:

- module boundaries aligned to doctrine
- tests aligned to doctrine
- explicit placeholders where semantics remain open

Exit condition:

Implementation follows doctrine instead of silently redefining it.
